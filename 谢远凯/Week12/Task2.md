前后端分离是现代软件开发中的一种架构模式。简单来说，就是让前端（负责页面展示）和后端（负责数据处理）彻底“分家”，各自独立开发、部署，通过 API 接口（通常是 JSON 格式）进行数据交互。'''## 历史对话如何存储- 存储位置(两套并行存储)  - 关系型数据库（用于UI/历史查看/管理）    - 会话信息和每条消息都保存到ChatSessionTable/ChatMessageTable  - Agent内部记忆（用于模型上下文/状态管理）    - 在每次chat调用时创建/使用AdvancedSQLiteSession(session_id=...,db_path=''),将agent的会话状态保存到conversations.db，供Agent在后续调用中加载历史。- 写入时机  - 新会话：services.chat.init_chat_session会在ChatSessionTable中建行，并插入一条system消息(content=get_init_message(task)).  - 每次收到用户输入时：在处理前会调用services.chat.append_message2db将user消息写入ChatMessageTable(关系DB)，随后运行Agent。  - 模型返回后：将拼接好的assistant回答再次通过services.chat.append_message2db写入关系DB。- 模型如何把历史作为下一次输入  - 保证调用时使用同给一个session id，这会被同时用于关系DB的会话查找和Agent的内存session。  - 在 chat() 中创建 AdvancedSQLiteSession(session_id=..., db_path="./assert/conversations.db") 并把这个 session 对象传给 Agent / Runner：services.chat.chat。Agent/Runner 会从该 AdvancedSQLiteSession 加载之前的对话上下文（上次保存的 agent memory / messages），并把这些历史作为 prompt / context 提给 LLM。  - 同时也把历史写入关系型DB（ChatMessageTable）以便前端通过 services.chat.get_chat_sessions 拉取并展示历史消息。- 确保前端请求携带并复用同一个 session_id（或由后端生成并返回），这样 AdvancedSQLiteSession 才能把历史上下文正确串联。参见生成 id 的函数：services.chat.generate_random_chat_id。文件：services/chat.py。- 关系 DB（ChatMessageTable）用于展示与审核，Agent 内部的 AdvancedSQLiteSession 用于提供给模型的上下文，两者必须同步（即 append_message2db 仍然要在合适时机写入），否则 UI 与模型上下文会不一致。存两份历史：一份关系型数据库用于给前端展示和管理（会话元信息、每条消息）；另一份是 agent 专用的本地会话存储，用于构建给模型的上下文记忆。关系库侧重可视化、审计和检索；agent 内存侧重快速加载、供模型使用的对话状态和工具调用历史。每次用户发起对话或模型返回回答时，都会把消息写入关系型数据库，确保前端能立即看到最新记录。同一个会话通过一个唯一 session_id 关联；只要复用这个 id，就能把关系库记录和 agent 内存串起来，形成连续的上下文。当需要让大模型继续对话时，agent 会从自己维护的会话存储中加载过去的消息和状态，把这些历史拼接进当前请求的上下文（prompt）里，作为模型的输入。若对话中用到外部工具，工具的调用和结果也会被写入 agent 的记忆，后续模型可以基于这些工具输出继续生成回答。因此前端要负责传回正确的 session_id；后端负责把新消息同时写入展示用的数据库和供模型使用的会话存储，从而保证 UI 展示与模型上下文一致。关键点：关系库用于展示/管理，agent 内存用于构建模型输入；两者并行写入并通过 session_id 关联以保证历史连续性。
