# 作业1 设计思路：LangChain 本地知识库问答系统

## 题目要求

基于 LangChain 框架，开发对本地知识库进行问答的逻辑，需要包含**文档检索 + LLM回答**两个核心流程。

## 整体架构

文档检索 + LLM回答 = RAG（Retrieval-Augmented Generation，检索增强生成）。

RAG 的核心理念是：LLM 的知识是训练时就固定下来的，无法回答训练数据之外的问题（比如公司内部文档、最新的信息）。RAG 通过先从外部知识库中检索相关信息，再把检索结果喂给 LLM 作为参考，让 LLM 基于这些资料来回答，解决了"LLM不知道私有知识"的问题。

整个流程分为5个环节：

```
用户问题 → [1.文档加载] → [2.文档分块] → [3.向量化存储]
                                                    ↓
用户回答 ← [5.LLM回答] ← [4.相似度检索] ←────── 用户提问
```

## 各环节设计思路

### 1. 文档加载（load_documents）

**做了什么**：从本地 docs 目录加载所有 .txt 文件。

**为什么这样写**：
- 使用 DirectoryLoader + TextLoader：LangChain 提供了统一的文档加载接口，后续如果要支持 PDF、Markdown 等其他格式，只需要替换 Loader 即可，其他代码不用改
- 检查目录是否存在并自动创建：首次运行时不会因为缺少目录而崩溃，给用户明确的提示
- 设置 encoding="utf-8"：中文文档必须指定编码，否则 Windows 上默认 GBK 会乱码

### 2. 文档分块（split_documents）

**做了什么**：用 RecursiveCharacterTextSplitter 把长文档切成 500 字左右的小块，块之间有 50 字的重叠。

**为什么这样写**：

*为什么要分块？*
- LLM 的上下文窗口有限，不能把整本书一次性塞进去
- 检索精度：小块检索更精准，不会把不相关的内容也拉进来
- 成本控制：只发送相关的小块给 LLM，节省 token

*为什么 chunk_size=500？*
- 500 字足够包含一个完整的语义段落（一个概念、一个知识点）
- 太小（如200字）会切断语义，太大（如1000字）会降低检索精度

*为什么 chunk_overlap=50？*
- 有重叠可以避免语义被切在块边界上
- 比如"Python的常用框架包括Django和Flask"这句话如果刚好被切在两块之间，下一块就失去了上文

*为什么 separators 用 ["\n\n", "\n", "。", "！", "？", ".", "!", "?", " "] ？*
- RecursiveCharacterTextSplitter 会按顺序尝试这些分隔符
- 优先在段落边界（\n\n）切分，其次在句子边界（。！？）切分，最后才在词边界切分
- 这样切出来的块语义更完整，不会在句子中间断掉

### 3. 向量化存储（create_vector_store）

**做了什么**：用 OpenAIEmbeddings 把每个文本块转换成一个向量，然后用 FAISS 建立向量索引。

**为什么这样写**：

*为什么要把文本转成向量？*
- 计算机不能直接比较两段文字"意思是否相近"，但可以计算两个向量之间的距离
- 语义相近的文本，其向量距离也近
- 用户提问 → 转成向量 → 在向量库中找距离最近的几个文本块 → 这就是最相关的资料

*为什么用 FAISS？*
- Facebook 开源的向量检索库，在本地运行，不需要额外的向量数据库服务
- 适合小规模知识库，速度快，部署简单
- 对于作业级别的知识库（几个文档、几十个块），FAISS 完全够用

*为什么用阿里云的 text-embedding-v3？*
- 中文嵌入效果好（对比 OpenAI 的 text-embedding-ada-002，对中文支持更好）
- 与 LLM 使用同一平台（阿里云 DashScope），统一管理

### 4. 构建检索问答链（build_qa_chain）

**做了什么**：构建一条 LangChain 链：从向量库检索相关文档 → 把检索结果和用户问题一起填入提示词模板 → 发给 LLM → 解析返回文本。

**为什么这样写**：

*为什么用 RunnablePassthrough？*
```python
{"context": retriever | format_docs, "question": RunnablePassthrough()}
```
- 这条链需要两个输入：context（检索到的文档）和 question（原始问题）
- retriever 从向量库取回相关文档，经过 format_docs 拼接成一段文字，赋值给 context
- RunnablePassthrough() 把用户输入原样透传给 question
- 这种声明式的写法让数据流一目了然，不需要写中间变量

*为什么 k=3（检索3条相关片段）？*
- 太少（k=1）：可能遗漏关键信息，降低回答质量
- 太多（k=10）：信息冗余，超出 LLM 的注意力范围，还可能引入噪声
- k=3 是经验值，平衡了信息覆盖和精度的要求

*为什么 Prompt 模板要这样设计？*
```
参考资料：
{context}

问题：{question}

回答：
```
- 先放参考资料，再放问题：LLM 会先"阅读"资料再"理解"问题
- 明确指示"如果参考资料中没有相关信息，请如实说明"：防止 LLM 编造答案（幻觉）
- 结构清晰，角色明确：知识库问答助手

### 5. 主流程（main）

**做了什么**：串联以上4个环节，然后进入交互式问答循环。

**为什么这样写**：

*为什么用交互式循环而不是一次性回答？*
- 知识库问答的典型使用场景是用户反复提问，每次都可能问不同的问题
- 如果一次性结束，每次提问都要重新加载文档、重新建立索引，非常低效
- 向量库只在启动时构建一次，后续每次提问只做检索+LLM回答，速度快

*为什么先显示检索结果再显示LLM回答？*
- 让用户能看到 LLM 是基于哪些资料回答的，增加可信度
- 如果检索结果就不对，说明知识库里没有相关内容，LLM 的回答自然不可信

*为什么用阿里云的通义千问（qwen-flash）？*
- 中文理解能力强
- qwen-flash 速度快、成本低，适合作业演示
- 通过 OpenAI 兼容接口调用，LangChain 的 ChatOpenAI 可以直接对接

## 技术选型总结

| 环节 | 选型 | 理由 |
|------|------|------|
| 文档加载 | LangChain DirectoryLoader | 统一接口，扩展方便 |
| 文档分块 | RecursiveCharacterTextSplitter | 按自然边界切分，语义完整 |
| 向量化 | text-embedding-v3 | 中文效果好 |
| 向量存储 | FAISS | 本地运行，轻量高效 |
| 大模型 | 通义千问 qwen-flash | 中文能力好，速度快 |
| 链组装 | LangChain LCEL | 声明式写法，数据流清晰 |
