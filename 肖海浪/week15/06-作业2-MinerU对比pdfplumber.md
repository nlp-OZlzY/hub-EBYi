## 一、MinerU2.5 论文核心要点

### 1.1 基本信息

| 项目 | 内容 |
|------|------|

| 模型参数 | 仅 1.2B（Vision Encoder 675M + LLM 0.5B） |
| 核心思想 | **解耦式两阶段推理**：全局布局分析 + 局部内容识别 |

---

### 1.2 核心架构：两阶段推理

MinerU 将文档解析拆为两步，而不是像传统 VLM 那样一次性处理整张图片：

```
输入文档
    │
    ▼
【阶段1：全局布局分析】     ← 下采样到 1036×1036 低分辨率，快速识别所有元素
    输出：元素bbox + 类别 + 阅读顺序
    │
    ▼
【阶段2：局部内容识别】     ← 从原始高分辨率图片裁剪各区域，原生分辨率识别
    输出：Markdown / JSON
```

**为什么这样设计？**
- 传统方法在高分辨率原图上做全局推理，计算量 O(N²) 非常大
- MinerU 先低分辨率扫一遍确定位置，再高分辨率精确识别内容，**快且准**

---

### 1.3 三大核心组件

| 组件 | 实现 | 参数 | 作用 |
|------|------|------|------|
| 视觉编码器 | NaViT-675M（基于 Qwen2-VL） | 675M | 支持动态分辨率图片输入 |
| Patch Merger | 两层 MLP | - | 压缩视觉 token，减少计算量 |
| 语言解码器 | Qwen2-Instruct-0.5B（改用 M-RoPE） | 0.5B | 生成 Markdown / JSON |

---

### 1.4 关键技术创新

| 创新点 | 说明 |
|--------|------|
| **M-RoPE（多维旋转位置编码）** | 让 0.5B 小模型也能处理不同宽高比的文档图片 |
| **NaViT + 窗口注意力** | 动态分辨率输入，GPU 显存开销降至传统方案的 1/8 |
| **OTSL 表格语言** | 将 HTML 的 28+ token 压缩至 5 个，表格识别效率大幅提升 |
| **ADR 公式识别框架** | 复合公式分解为原子公式逐行识别，再结构化重组 |
| **IMIC 难样本挖掘** | 多模型交叉推理，不一致的样本标记为困难样本重点训练 |
| **PageIoU 评估指标** | 比传统 IoU 更符合文档视觉感知的像素级评估 |

### 1.6 MinerU2.5 vs MinerU2.5-Pro 的区别

MinerU2.5-Pro 的核心升级：**不改一行模型架构，纯靠数据工程驱动**。

## 二、MinerU 使用方法

### 2.1 安装

```bash
pip install magic-pdf[full]
```

### 2.2 命令行使用

```bash
# 基础解析
mineru -p input.pdf -o output_dir

# GPU加速
mineru -p input.pdf -o output_dir -b vlm-http-client -u http://127.0.0.1:30000
```

### 2.3 Python API

```python
from magic_pdf.pipe.UNIPipe import UNIPipe
from magic_pdf.rw.DiskReaderWriter import DiskReaderWriter

# 创建解析管道
pipe = UNIPipe(pdf_bytes, jso_useful_key={}, image_writer=DiskReaderWriter("./output"))

# 解析
pipe.pipe_classify()
pipe.pipe_analyze()  # 版面分析
pipe.pipe_parse()    # 内容解析

# 输出
md_content = pipe.pipe_mk_markdown("./output/images")
```

### 2.4 输出格式

```
output/
├── {pdf_name}/
│   ├── auto/              # 自动模式结果
│   │   ├── images/        # 提取的图片
│   │   └── content.md     # Markdown 文件
│   └── layout.pdf         # 版面分析可视化
```

## 三、MinerU vs pdfplumber 对比

### 3.1 对比概览

| 对比维度 | pdfplumber | MinerU |
|----------|-----------|--------|
| **技术原理** | 纯规则解析（直接读取PDF内部字符/线条/矩形坐标） | 深度学习（VLM视觉理解 + OCR） |
| **文本提取** | 提取准确（仅限原生文本型PDF） | 提取准确（支持扫描件OCR） |
| **扫描件PDF** | 不支持（需要文本层） | 支持（109种语言OCR） |
| **图片提取** | 支持提取嵌入图片 | 自动识别并提取文档中的图表 |
| **表格识别** | 通过线条检测表格（无框线表格识别差） | 深度学习识别表格（支持无线表、合并单元格） |
| **公式识别** | 乱码或丢弃 | 识别为可编译的LaTeX格式 |
| **版式保持** | 弱（多栏布局阅读顺序错乱） | 强（自动识别并重排阅读顺序） |
| **页眉页脚** | 混入正文 | 自动识别并过滤 |
| **处理速度** | 快（纯CPU，秒级） | 较慢（需GPU，约2页/秒） |
| **资源需求** | 低（无需GPU） | 高（需要GPU） |
| **安装难度** | 简单（pip install） | 较复杂（需下载模型权重） |
| **输出格式** | 纯文本 | Markdown / JSON |

### 3.5 各维度详细对比

#### 3.5.1 文本提取

| 场景 | pdfplumber | MinerU |

## 四、选型建议

| 场景 | 推荐 | 原因 |
|------|------|------|
| 学术论文、技术报告 | MinerU | 公式LaTeX、表格还原、多栏布局 |
| 中文文档（财报/合同/研报） | MinerU | 中文支持优秀 |
| RAG知识库构建 | MinerU | Markdown输出 + 高召回率 |
| 扫描件PDF | MinerU | pdfplumber无法处理 |
| 简单单栏纯文本PDF | pdfplumber | 轻量快速 |
| 仅需提取表格数据 | pdfplumber | API简洁 |
| 无GPU环境 | pdfplumber | 无需GPU |
| 快速批量处理 | pdfplumber | 纯CPU，速度快 |

## 五、总结

**pdfplumber** 是一个轻量级工具，适合处理结构简单的原生PDF，利用PDF内部的字符坐标信息提取文本和表格。优点是速度快、部署简单，缺点是只能处理有文本层的PDF，对扫描件无效，对多栏布局和无线表格支持差。

**MinerU** 是一个基于深度学习的文档解析模型，采用解耦式两阶段架构，先进行全局版面分析再进行局部内容识别。优点是支持扫描件OCR、公式LaTeX输出、复杂表格还原、版面自动识别，输出高质量的Markdown格式；缺点是需要GPU资源，部署相对复杂，处理速度不如pdfplumber快。

**在实际项目中**，可以结合两者：简单PDF用pdfplumber快速处理，复杂PDF（学术论文、扫描件、中文文档）用MinerU保证质量。
