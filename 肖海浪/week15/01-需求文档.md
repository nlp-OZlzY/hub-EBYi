# 多模态RAG聊天机器人 - 需求文档

## 一、项目概述

### 1.1 项目背景

在信息爆炸的时代，大量知识存在于图文混排的PDF文档中。传统问答系统只能处理纯文本，无法理解图表、图片等视觉信息。本项目构建一个多模态RAG（检索增强生成）系统，支持用户上传PDF文档，系统自动解析并建立索引，用户可通过自然语言提问，系统检索相关内容（文本+图片）并生成回答。

### 1.2 项目目标

| 目标 | 说明 |
|------|------|
| 文档解析 | 支持PDF/DOCX/TXT文件上传，自动解析为结构化内容 |
| 多模态检索 | 支持文本和图片的向量检索 |
| 智能问答 | 基于检索结果，使用大模型生成准确回答 |
| 可扩展性 | 架构支持后续接入更多文档类型和模型 |

### 1.3 术语说明

| 术语 | 解释 |
|------|------|
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| Embedding | 将文本/图片转换为向量表示 |
| Milvus | 开源向量数据库，用于存储和检索向量 |
| Kafka | 消息队列，用于异步任务传递 |
| MinerU | 文档解析工具，将PDF转换为Markdown+图片 |
| bge | 文本Embedding模型，将文本转为512维向量 |
| CLIP | 多模态模型，将文本和图片转为1024维向量 |
| Qwen-VL | 阿里云多模态大模型，支持图文理解 |

## 二、核心需求

### 2.1 需求总览

| 需求ID | 功能模块 | 需求描述 | 优先级 |
|--------|---------|---------|--------|
| R01 | 文件上传 | 用户上传PDF/DOCX/TXT文件，保存到本地，发送Kafka消息触发解析 | P0 |
| R02 | 文档解析 | Worker消费Kafka消息，调用mineru解析PDF为markdown+图片 | P0 |
| R03 | 向量编码 | 使用bge编码文本、clip编码图片，存入Milvus向量数据库 | P0 |
| R04 | 多模态检索 | 用户提问，bge编码后在Milvus中检索相关文本和图片 | P0 |
| R05 | 多模态问答 | 将检索结果+用户问题发送给Qwen-VL生成回答 | P0 |
| R06 | 文件管理 | 查询/删除已上传文件，删除时清理向量数据 | P1 |

## 三、接口详细定义

### 3.1 POST /upload/document

**请求**：
```
Content-Type: multipart/form-data
Body: file (UploadFile) - 支持 .pdf / .docx / .txt
```

**成功响应** (200)：
```json
{
  "code": 200,
  "data": {
    "file_id": 1,
    "filename": "example.pdf",
    "filestate": "已上传"
  }
}
```

**错误场景**：
| 场景 | HTTP状态码 | 错误信息 |
|------|-----------|---------|
| 文件类型不支持 | 400 | "仅支持 pdf/docx/txt 格式" |
| 文件为空 | 400 | "文件不能为空" |
| 保存失败 | 500 | "文件保存失败" |

**业务规则**：
- 文件保存路径：`uploads/{uuid}.{ext}`
- 文件状态初始为"已上传"，解析完成后更新为"已解析"
- Kafka消息格式：`{"file_name": str, "file_path": str, "id": int}`

---

### 3.2 POST /chat

**请求**：
```json
{
  "question": "这张图片里左边那个设备的功能是什么？"
}
```

**成功响应** (200)：
```json
{
  "code": 200,
  "data": {
    "answer": "根据资料显示，左边的设备是...",
    "sources": [
      {"file_name": "doc1.pdf", "relevance": 0.85},
      {"file_name": "doc2.pdf", "relevance": 0.72}
    ]
  }
}
```

**业务规则**：
- 检索Top-5相关文档片段
- 回答必须基于检索到的资料，不能编造
- 如果资料中包含图片链接，需在回答中保留原始链接

---

### 3.3 GET /files

**成功响应** (200)：
```json
{
  "code": 200,
  "data": [
    {
      "id": 1,
      "filename": "example.pdf",
      "filepath": "uploads/abc123.pdf",
      "filestate": "已上传"
    }
  ]
}
```

---

### 3.4 DELETE /files/{file_id}

**成功响应** (200)：
```json
{
  "code": 200,
  "message": "删除成功"
}
```

**错误场景**：
| 场景 | HTTP状态码 | 错误信息 |
|------|-----------|---------|
| 文件不存在 | 404 | "文件不存在" |

**业务规则**：
- 删除SQLite记录
- 删除本地文件
- 清理Milvus中对应db_id的所有向量

## 四、数据模型

### 4.1 File表（SQLite）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | Integer | PK, 自增 | 文件ID |
| filename | String(255) | NOT NULL | 原始文件名 |
| filepath | String(1000) | NOT NULL | 本地存储路径 |
| filestate | String(20) | NOT NULL | 状态：已上传/解析中/已解析/解析失败 |

### 4.2 向量数据（Milvus Collection: rag_data_new）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Int64 | 主键，自增 |
| text_vector | FloatVector(512) | bge编码的文本向量 |
| clip_text_vector | FloatVector(1024) | clip编码的文本向量 |
| clip_image_vector | FloatVector(1024) | clip编码的图片向量 |
| text | VarChar | 原始文本chunk |
| db_id | Int64 | 关联File表的id |
| file_name | VarChar | 文件名 |
| file_path | VarChar | 文件路径 |

## 五、核心业务流程

### 5.1 上传流程
```
用户上传文件
  → 校验文件类型和大小
  → 保存到 uploads/ 目录
  → 写入SQLite（状态：已上传）
  → 发送Kafka消息到 topic "rag-data"
  → 返回文件信息
```

### 5.2 文档解析流程（Worker）
```
消费Kafka消息
  → 调用 mineru 解析PDF为markdown+图片
  → 文本切分为chunks（每块≤256字符）
  → 对每个chunk：
      → bge编码文本 → text_vector
      → clip编码文本 → clip_text_vector
      → clip编码图片 → clip_image_vector
  → 批量写入Milvus
  → 更新SQLite状态为"已解析"
```

### 5.3 问答流程
```
接收用户问题
  → bge编码问题
  → Milvus中检索Top-5相似文本（anns_field=text_vector）
  → 拼接检索结果为上下文
  → 调用Qwen-VL生成回答
  → 返回答案+来源信息
```

## 六、非功能性需求

| 维度 | 要求 |
|------|------|
| 并发 | 上传接口支持10并发 |
| 超时 | mineru解析单文件超时600秒 |
| 容错 | 解析失败不影响其他文件处理 |
| 日志 | 关键步骤需打印日志（上传、解析、检索、问答） |
| 存储 | 文件存储使用UUID命名，避免冲突 |
| 安全 | 文件类型白名单校验，文件大小限制50MB |
