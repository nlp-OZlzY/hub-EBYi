# 多模态RAG聊天机器人 - 架构设计文档

## 一、项目概述

### 1.1 项目背景

在信息爆炸的时代，大量知识存在于图文混排的PDF文档中。传统问答系统只能处理纯文本，无法理解图表、图片等视觉信息。本项目构建一个多模态RAG（检索增强生成）系统，支持用户上传PDF文档，系统自动解析并建立索引，用户可通过自然语言提问，系统检索相关内容（文本+图片）并生成回答。

### 1.2 项目目标

| 目标 | 说明 |
|------|------|
| 文档解析 | 支持PDF/DOCX/TXT文件上传，自动解析为结构化内容 |
| 多模态检索 | 支持文本和图片的向量检索 |
| 智能问答 | 基于检索结果，使用大模型生成准确回答 |
| 可扩展性 | 架构支持后续接入更多文档类型和模型 |

### 1.3 术语说明

| 术语 | 解释 |
|------|------|
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| Embedding | 将文本/图片转换为向量表示 |
| Milvus | 开源向量数据库，用于存储和检索向量 |
| Kafka | 消息队列，用于异步任务传递 |
| MinerU | 文档解析工具，将PDF转换为Markdown+图片 |
| bge | 文本Embedding模型，将文本转为512维向量 |
| CLIP | 多模态模型，将文本和图片转为1024维向量 |
| Qwen-VL | 阿里云多模态大模型，支持图文理解 |

### 3.2 各模块职责说明

#### 3.2.1 路由层 (api/)

**职责**：接收HTTP请求，参数校验，调用Service层，返回响应

| 文件 | 接口 | 职责 |
|------|------|------|
| `upload.py` | POST /upload/document | 接收文件，校验类型和大小，调用FileService |
| `chat.py` | POST /chat | 接收问题，调用ChatService，返回答案 |
| `files.py` | GET /files, DELETE /files/{id} | 文件列表查询和删除 |

**代码示例**：
```python
# api/upload.py
from fastapi import APIRouter, UploadFile, HTTPException
from services.file_service import FileService
from models.schemas import UploadResponse, ErrorResponse

router = APIRouter()
file_service = FileService()

@router.post("/upload/document", response_model=UploadResponse, responses={400: {"model": ErrorResponse}})
async def upload_document(file: UploadFile):
    """上传文档接口"""
    # 1. 校验文件类型
    if not file.filename.endswith(('.pdf', '.docx', '.txt')):
        raise HTTPException(status_code=400, detail="仅支持 pdf/docx/txt 格式")

    # 2. 校验文件大小
    content = await file.read()
    if len(content) == 0:
        raise HTTPException(status_code=400, detail="文件不能为空")

    # 3. 调用Service处理
    result = file_service.save_and_publish(file.filename, content)

    return UploadResponse(data=result)
```

#### 3.2.2 服务层 (services/)

**职责**：实现业务逻辑，不处理HTTP细节

| 文件 | 类 | 职责 | 依赖 |
|------|-----|------|------|
| `file_service.py` | FileService | 文件保存、记录创建、Kafka消息发送 | ORM, Kafka |
| `chat_service.py` | ChatService | 编码问题、检索、调用LLM | SearchService, LLM |
| `search_service.py` | SearchService | Milvus向量检索 | Milvus |
| `encode_service.py` | EncodeService | bge/clip编码 | 模型 |

**代码示例**：
```python
# services/file_service.py
import os
import uuid
from kafka import KafkaProducer
from orm_model import File, Session

class FileService:
    def __init__(self):
        self.upload_dir = "uploads"
        self.producer = KafkaProducer(
            bootstrap_servers="localhost:9092",
            value_serializer=lambda v: json.dumps(v).encode('utf-8')
        )

    def save_and_publish(self, filename: str, content: bytes) -> dict:
        """保存文件并发送Kafka消息"""
        # 1. 生成唯一文件名
        ext = os.path.splitext(filename)[1]
        save_name = f"{uuid.uuid4()}{ext}"
        save_path = os.path.join(self.upload_dir, save_name)

        # 2. 保存文件
        with open(save_path, "wb") as f:
            f.write(content)

        # 3. 创建数据库记录
        with Session() as session:
            record = File(
                filename=filename,
                filepath=save_path,
                filestate="已上传"
            )
            session.add(record)
            session.commit()
            file_id = record.id

        # 4. 发送Kafka消息
        self.producer.send("rag-data", value={
            "file_name": filename,
            "file_path": save_path,
            "id": file_id
        })
        self.producer.flush()

        return {"file_id": file_id, "filename": filename, "filestate": "已上传"}
```

## 五、数据流设计

### 5.1 上传流程数据流

```
用户选择文件
    │
    ▼
浏览器发送POST /upload/document
    │
    ▼
FastAPI接收请求
    │
    ├─▶ 校验文件类型 (.pdf/.docx/.txt)
    │
    ├─▶ 校验文件大小 (≤50MB)
    │
    ├─▶ 保存文件到 uploads/{uuid}.{ext}
    │
    ├─▶ 创建SQLite记录 (filestate="已上传")
    │
    ├─▶ 发送Kafka消息到 topic "rag-data"
    │   消息格式: {"file_name": "...", "file_path": "...", "id": 1}
    │
    └─▶ 返回 {file_id, filename, filestate}
```

---

### 5.2 离线解析流程数据流

```
Kafka Consumer 监听 topic "rag-data"
    │
    ▼
收到消息
    │
    ├─▶ 更新SQLite状态为 "解析中"
    │
    ├─▶ 调用 MinerU 解析PDF
    │   输入: uploads/abc123.pdf
    │   输出: processed/abc123/vlm/content.md + images/
    │
    ├─▶ 读取解析后的Markdown文件
    │
    ├─▶ 文本切分 (split_text2chunks)
    │   输入: 长文本
    │   输出: [chunk1, chunk2, ...] (每块≤256字符)
    │
    ├─▶ 对每个chunk进行编码
    │   ├─ bge编码文本 → text_vector (512维)
    │   ├─ clip编码文本 → clip_text_vector (1024维)
    │   └─ clip编码图片 → clip_image_vector (1024维)
    │
    ├─▶ 写入Milvus
    │   数据: {text_vector, clip_text_vector, clip_image_vector, text, db_id, file_name, file_path}
    │
    └─▶ 更新SQLite状态为 "已解析"
```

---

### 5.3 问答流程数据流

```
用户输入问题
    │
    ▼
浏览器发送POST /chat
    │
    ▼
FastAPI接收请求
    │
    ├─▶ bge编码问题 → question_vector (512维)
    │
    ├─▶ Milvus检索
    │   输入: question_vector
    │   参数: anns_field="text_vector", limit=5
    │   输出: Top-5相似文档片段
    │
    ├─▶ 拼接Prompt
    │   模板: "基于资料回答问题：{question}\n\n相关资料：{context}"
    │
    ├─▶ 调用Qwen-VL生成回答
    │   输入: Prompt
    │   输出: 自然语言回答
    │
    └─▶ 返回 {answer, sources}
```

## 六、数据库设计

### 6.1 SQLite - 文件表 (files)

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | INTEGER | PRIMARY KEY, AUTOINCREMENT | 文件唯一标识 |
| filename | VARCHAR(255) | NOT NULL | 原始文件名 |
| filepath | VARCHAR(1000) | NOT NULL | 服务器存储路径 |
| filestate | VARCHAR(20) | NOT NULL | 处理状态 |

**状态值说明**：
| 状态值 | 含义 |
|--------|------|
| 已上传 | 文件已保存，等待解析 |
| 解析中 | Worker正在处理 |
| 已解析 | 解析完成，向量已入库 |
| 解析失败 | 处理异常 |

---

### 6.2 Milvus - 向量表 (rag_data_new)

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | INT64 | 主键，自增 |
| text_vector | FLOAT_VECTOR(512) | bge编码的文本向量 |
| clip_text_vector | FLOAT_VECTOR(1024) | clip编码的文本向量 |
| clip_image_vector | FLOAT_VECTOR(1024) | clip编码的图片向量 |
| text | VARCHAR(65535) | 原始文本chunk |
| db_id | INT64 | 关联files表的id |
| file_name | VARCHAR(512) | 文件名 |
| file_path | VARCHAR(1024) | 文件路径 |

| 选型 | 理由 |
|------|------|

| Milvus | 专业向量数据库，支持多种索引，性能好 |
| Kafka | 高吞吐，支持持久化，适合异步任务场景 |
| MinerU | 中文PDF解析效果好，支持图文混排 |
| bge | 中文文本编码效果优秀，模型轻量 |
| CLIP | 支持文本和图片统一编码，便于跨模态检索 |

## 十、异常处理设计

### 10.1 异常分类

| 异常类型 | 处理方式 | 示例 |
|---------|---------|------|
| 参数校验异常 | 返回400，提示用户 | 文件类型不支持、问题为空 |
| 业务逻辑异常 | 返回400/404，提示用户 | 文件不存在 |
| 外部服务异常 | 返回500，记录日志 | Milvus连接失败、LLM调用超时 |
| 系统异常 | 返回500，记录日志 | 磁盘满、内存不足 |

---

### 10.2 异常处理流程

```python
# 统一异常处理
from fastapi import HTTPException
from fastapi.responses import JSONResponse

@app.exception_handler(HTTPException)
async def http_exception_handler(request, exc):
    return JSONResponse(
        status_code=exc.status_code,
        content={"code": exc.status_code, "message": exc.detail}
    )

@app.exception_handler(Exception)
async def general_exception_handler(request, exc):
    logger.error(f"Unexpected error: {exc}")
    return JSONResponse(
        status_code=500,
        content={"code": 500, "message": "服务器内部错误"}
    )
```

## 十一、安全设计

### 11.1 安全措施

| 风险点 | 措施 |
|--------|------|
| 文件上传漏洞 | 限制文件类型、大小，文件名使用UUID |
| SQL注入 | 使用ORM，不拼接SQL |
| 接口未授权 | 后续可接入JWT认证 |
| 敏感信息泄露 | 配置文件使用环境变量 |
